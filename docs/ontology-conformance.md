# SBOL schema conformance regression

`sbol-rs` pins a copy of the upstream SBOL 3 OWL document published by
[SynBioDex/sbol-owl3](https://github.com/SynBioDex/sbol-owl3) and uses
it as a regression fixture against the IRIs declared in
`crates/sbol3/src/vocab.rs`. The check runs as a normal integration
test (`crates/sbol3/tests/sbol_owl3_conformance.rs`), so any drift
(in either direction) fails CI rather than going unnoticed until a
user files a bug.

This system is intentionally *narrow*: it asserts agreement on IRI
membership, not on domain/range/cardinality restrictions. Treat the
OWL as a cross-check on the SBOL 3.1.0 PDF, which remains the
authoritative source.

The current pinned-state report (counts, allowlist contents, any
drift) is committed at
[`docs/sbol-owl3-conformance.md`](sbol-owl3-conformance.md) and
regenerated by
`cargo run -p sbol3 --bin generate-sbol-owl3-conformance-report`.
A freshness test (`crates/sbol3/tests/sbol_owl3_conformance_report.rs`)
fails CI if the committed file falls out of date.

## What the test checks

1. **Pin integrity.** The fixture's sha256 must match the value
   recorded in `tests/fixtures/sbol-owl3/manifest.toml`. Hand-editing
   the fixture without updating the manifest fails this check first.
2. **OWL ⊆ Rust (modulo allowlist).** Every `http://sbols.org/v3#X`
   IRI declared in the pinned OWL as an `owl:Class`,
   `owl:ObjectProperty`, or `owl:DatatypeProperty` must appear as a
   string constant in `vocab.rs`, or carry an entry in
   `OWL_ONLY_ALLOWLIST` with a one-line rationale.
3. **Rust ⊆ OWL (modulo allowlist).** Every `http://sbols.org/v3#X`
   IRI in `vocab.rs` must appear in the pinned OWL, or carry an
   entry in `RUST_ONLY_ALLOWLIST` with a citation to the SBOL 3.1.0
   spec section that legitimizes it. The `#zero` bug fixed in the
   `0.1.0 → Unreleased` window is the reference case for "vocab.rs
   IRI not in OWL **and** not in the spec."
4. **Shape floor.** A loose lower bound on the OWL's class /
   property counts catches silently-truncated downloads.

The allowlists also catch their own staleness: if an OWL-only entry
disappears from upstream, or a Rust-only entry appears in upstream,
the test fails so the allowlist can be cleaned up.

## When the test fails

- **Pin integrity failure** → the fixture was hand-edited or partially
  overwritten. Run the update tool below to re-pin, or revert.
- **Missing from `vocab.rs`** → upstream added a new SBOL term and we
  have not surfaced it. Either add the constant, or (if the new term
  is an abstract super-property or umbrella class we deliberately
  ignore) add it to `OWL_ONLY_ALLOWLIST` with a rationale.
- **Missing from `sbol3.rdf`** → vocab.rs declares an IRI the
  pinned OWL omits. Two possibilities:
  - The OWL is incomplete (common: it does not declare every
    SBOL value enumeration). File upstream, add the IRI to
    `RUST_ONLY_ALLOWLIST` with a spec citation.
  - The IRI is **not in the spec**. Delete the constant and the
    validation code that consumed it (see the `#zero` removal in
    `CHANGELOG.md` for the procedure).

## Refreshing the pin

```sh
cargo run -p sbol-ontology --bin update-sbol-owl3-fixture
```

The tool fetches the current `main` of `SynBioDex/sbol-owl3`,
downloads `sbol3.rdf` at that commit, recomputes the sha256, and
rewrites `manifest.toml`. Then:

```sh
cargo test -p sbol3 --test sbol_owl3_conformance
```

If the fixture changed semantically, the OWL-side coverage test
will surface new diffs. Triage each one into the allowlist or fix
the underlying drift, then commit the updated fixture, manifest,
and allowlist entries in a single commit so the trail is auditable.

## What this system does not do

- It does **not** compare `rdfs:domain` / `rdfs:range` /
  `owl:Restriction` declarations against the Rust data model.
  That comparison is more powerful but materially harder; it would
  require a second extraction pass plus a corresponding "expected
  shape" table in the test. It is a follow-up if the IRI-level
  regression proves valuable.
- It does **not** validate user-supplied SBOL documents against the
  OWL. The runtime validator (`Document::validate`) consults the
  SBOL 3.1.0 rule catalog, not the OWL. The OWL is a cross-check on
  our rule catalog, not a runtime input.
- It does **not** fetch the OWL at test time. The pin is committed
  to the repo so the test runs offline and deterministically; the
  update tool is the only network-bound part of the system.

## Why this exists

The `#zero` bug (`vocab.rs` exporting a `CARDINALITY_ZERO` constant
for an IRI that is not in SBOL 3.1.0 Table 14) went undetected
because no fixture exercised it. Comparing against the upstream OWL
would have surfaced the drift on day one. The regression system is
designed so that the next such drift fails CI on the same commit
that introduces it.
