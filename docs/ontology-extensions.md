# Ontology extensions

`sbol-ontology` bundles a curated set of ontology snapshots that the
validator queries offline: EDAM, SBO, SO, GO, ChEBI, and Cell Ontology.
Larger ontologies (NCIT in particular) would more than triple the
compiled crate size, so they ship as **runtime extensions** that you
install once into a local cache and then opt in to per validation.

This document is for library and CLI users. For an overview of how the
validator decides whether a term is recognized, see
[`docs/validation.md`](validation.md).

## At a glance

- **Bundled** (always on, no setup): EDAM, SBO, SO, GO, ChEBI, CL.
- **Extension** (install once, opt in): NCIT today; future
  domain-specific ontologies and user-supplied TSVs.
- **Validation never touches the network.** Installs and downloads
  happen only through explicit `cache.install` / `cache.ensure_installed`
  calls (Rust) or `sbol ontology install` (CLI).
- **Precedence**: bundled terms always win on conflicts. Extensions
  can add new terms and parent links but cannot rewrite bundled facts.

## Where the cache lives

Resolution order:

1. `$SBOL_ONTOLOGY_CACHE` if set.
2. `$XDG_CACHE_HOME/sbol/ontologies/` on Linux.
3. `$HOME/.cache/sbol/ontologies/` (Linux fallback).
4. `$HOME/Library/Caches/sbol/ontologies/` on macOS.
5. `%LOCALAPPDATA%\sbol\ontologies\` on Windows.

Each installed extension is two files: `<name>.tsv` (the compact fact
table, format-versioned with `# format_version: 1`) and `<name>.manifest`
(source URL, sha256 of the upstream payload, sha256 of the generated
TSV, install timestamp).

## CLI workflow

```sh
# Where does the cache live on this machine?
sbol ontology path

# Download NCIT, scope it to the spec-recommended subtrees, and install
# into the cache. Idempotent: re-running is a no-op unless --force.
sbol ontology install ncit

# Or force a re-download:
sbol ontology install ncit --force

# What's installed?
sbol ontology list

# Re-hash to detect tampering / partial downloads:
sbol ontology verify
sbol ontology verify ncit

# Drop an installed extension:
sbol ontology remove ncit
```

Once installed, opt in per validation:

```sh
sbol validate doc.ttl --ontology ncit
```

`--ontology` is repeatable and precedence runs left-to-right (the
bundled snapshot still wins on every conflict).

## Rust API

```rust
use sbol::{Document, ValidationOptions};
use sbol_ontology::{KnownOntology, OntologyCache};

// Default cache path. Performs no IO.
let cache = OntologyCache::default();

// Download + build only when the cache entry is missing. Safe to call
// from library startup; idempotent on the second run.
let manifest = cache.ensure_installed(KnownOntology::Ncit.descriptor())?;
println!("ncit installed at {} (sha256 {})", manifest.installed_at, manifest.fact_sha256);

// Load the now-installed extension as a regular `Ontology`.
let ncit = cache.load(KnownOntology::Ncit.name())?;

// Attach it to a validation run. Bundled facts remain authoritative;
// the extension is consulted when bundled queries return `None`.
let options = ValidationOptions::default().with_ontology_extension(ncit);
let report = document.validate_with(options);
```

For users who manage their own TSVs (air-gapped deployments,
lab-specific ontologies generated by other tooling), skip the cache
and load directly:

```rust
use sbol_ontology::Ontology;

let lab = Ontology::from_tsv_path("/path/to/lab-ontology.tsv")?;
let options = ValidationOptions::default().with_ontology_extension(lab);
```

`OntologyCache` is also safe to construct at a custom path with
[`OntologyCache::at`] when the default XDG location is not appropriate.

## NCIT scope

The `KnownOntology::Ncit` descriptor scopes the NCIT install to the
five subtrees the SBOL 3.1.0 specification calls out by example in
§7.9.2 (Components representing experimental reagents):

- `NCIT:C12508`: Cell (cell-line roles)
- `NCIT:C14419`: Organism Strain
- `NCIT:C85504`: Growth Medium
- `NCIT:C64356`: Positive Control and siblings
- `NCIT:C120268`: Inducer

Other NCIT terms are deliberately excluded. The full NCIT graph is
over 180,000 terms and adds little SBOL-specific value beyond the
experimental-reagent context. If you have a use case that needs broader
coverage, construct your own [`OntologyDescriptor`] and call
[`OntologyCache::install`] with it.

## Custom extensions

To ship a lab-specific ontology, write a TSV that follows the bundled
schema (header column row plus `term` / `branch` / `compat` /
`conflict` / `component_role` / `component_role_compat` /
`participation_compat` lines) and prefix it with `# format_version: 1`.
Drop it in the cache directory as `<name>.tsv` with a sibling
`<name>.manifest`, or load it directly with
`Ontology::from_tsv_path("...")`.

The TSV schema is treated as a stable public contract; semver bumps
of `sbol-ontology` only change the `format_version` if the schema
itself changes incompatibly.

## Reproducibility caveat

Two users on the same `sbol` version can observe different validation
reports when their caches contain different extension snapshots, or
different versions of the same extension. The bundled snapshot is
pinned to the release. Extensions are pinned to whatever the user
last installed. For deterministic CI, either:

- Commit a `*.tsv` file to your repo and load it with
  `Ontology::from_tsv_path`, or
- Pin the install time and source-checksum and reject mismatches with
  `OntologyCache::verify`.
