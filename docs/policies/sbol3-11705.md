# sbol3-11705: Orientation constraint contradictions

**Status:** Partial  •  **Spec section:** §6.4.3  •  **PolicyOptions knob:** `orientation_reasoning`

## Spec wording

> The restriction values of a Constraint MUST satisfy the orientation relations they assert; in particular, `sameOrientationAs` and `oppositeOrientationOf` MUST NOT both hold for the same pair of Features.

## Ambiguity

The spec's "MUST NOT both hold" implies a closure step the validator may not be able to take: orientation reasoning across chains of constraints is implementation-defined.

## Decision

Default `Conservative`: emit error only on directly-stated contradictions (one Constraint asserting both relations on the same pair, or two Constraints with conflicting direct assertions). `Strict` performs a transitive closure. `Lenient` drops the check.

## Rationale

Closure reasoning over arbitrary constraint graphs is undecidable in general. Direct contradictions catch the common authoring mistake; closure is for advanced workflows.

## Examples

### Conservative (default)
Matches today's validator behavior: the diagnostic shape and severity are exactly what the current test suite expects.

### Strict
(promotes the warning to an error, or activates a check that is currently suppressed.)

### Lenient
(drops the check; the rule is recorded in coverage as suppressed.)
