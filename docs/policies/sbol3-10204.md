# sbol3-10204: Provenance generation cycles

**Status:** Implemented (same-document)  •  **Spec section:** §6.1  •  **PolicyOptions knob:** *(none; policy is baked into the algorithm)*

## Spec wording

> Provenance history formed by `prov:wasGeneratedBy` properties of
> Identified objects and `prov:entity` references in `prov:Usage`
> objects MUST NOT form circular reference chains.

## Ambiguity

Two interpretive gaps in the spec wording:

1. **Cross-document scope.** "Provenance history" naturally crosses
   documents: an Activity defined in one document can reference an
   entity in another. A pure local-graph traversal misses cycles whose
   path leaves and re-enters the document.
2. **Length-1 self-cycles.** A strict reading rejects the chain
   `X -wasGeneratedBy-> A -qualifiedUsage-> U -entity-> X`. The
   canonical SBOLTestSuite fixture
   (`SBOL3/provenance_entity/activity/activity.ttl`) exercises this
   exact pattern: a `codon_optimization_activity` produces
   `toggle_switch_optimised` and lists the same entity in a
   `Usage` with `role = sbol:design`. libSBOLj3 round-trips and
   validates this fixture without flagging the cycle.

## Decision

- **Cross-document scope is deferred.** The implemented algorithm
  walks only within a single `Document`. A resolver-aware version that
  walks across a `DocumentSet` is tracked in the implementation
  roadmap (Tier 3) and may land when there's a concrete user need.
- **Length-1 self-cycles are tolerated.** The cycle detector flags a
  revisit only when the path has crossed at least one other distinct
  entity. A direct `X -> A -> Usage -> X` is allowed; an indirect
  `X -> A1 -> ... -> Y -> A2 -> ... -> X` is rejected.

## Rationale

The length-1 carve-out aligns with libSBOLj3 (the only other
maintained SBOL 3 reference implementation) and with the
SBOLTestSuite, which treats this pattern as valid PROV-O usage. The
SBOL community treats `prov:Usage` with workflow roles (design /
build / test / learn) as a way to *tag* an activity's outputs, not as
a strict pre-input-of relationship. Forbidding the self-loop would
require rewriting the canonical test fixture and disagreeing with the
Java reference.

Cross-document deferral is a cost/value decision: the resolver-aware
walk requires a multi-document traversal abstraction that doesn't
exist yet, and no user has reported hitting a multi-document
provenance cycle in practice. Same-document coverage takes the rule
from `Unimplemented` to `Error` and closes the catalog at 109/109
machine-checkable rules.

## Examples

### Conservative (default)

The validator emits `sbol3-10204` at `Error` severity when the
generation chain forms a cycle through two or more distinct
Identified entities within the document. Single-activity self-loops
are allowed; cycles involving an entity not present in the document
are silently traversed past (the missing entity short-circuits the
walk).

### Strict

Not implemented. A strict mode would reject single-activity
self-loops and require all referenced entities to be present in the
document. Re-introduce when a user needs the stricter behavior.

### Lenient

Not implemented. A lenient mode would suppress the rule entirely.
Achievable today via `ValidationOptions::allow("sbol3-10204")`.
