# sbol3-12806: Hash algorithm token registry

**Status:** Partial  •  **Spec section:** §6.10  •  **PolicyOptions knob:** `hash_algorithm_registry`

## Spec wording

> An Attachment's hashAlgorithm MUST identify a known hash algorithm.

## Ambiguity

MUST-rule. "Known" is not defined; the spec doesn't pin a registry.

## Decision

Default `Conservative`: accept any registry-style token (lowercase ASCII, hyphens, digits). `Strict` (`KnownAlgorithm`) restricts to a curated allowlist (sha2-256, sha3-256, blake3, etc.). `Lenient` skips the check.

## Rationale

Registry-style is the broadest defensible accept; strict whitelisting is for environments that have committed to a hash policy.

## Examples

### Conservative (default)
Matches today's validator behavior: the diagnostic shape and severity are exactly what the current test suite expects.

### Strict
(promotes the warning to an error, or activates a check that is currently suppressed.)

### Lenient
(drops the check; the rule is recorded in coverage as suppressed.)
