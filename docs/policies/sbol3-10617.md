# sbol3-10617: Same-encoding sequence element conflicts

**Status:** Partial  •  **Spec section:** §6.4  •  **PolicyOptions knob:** `conflict_resolution`

## Spec wording

> If two Sequence objects have the same encoding, then their elements properties MUST have identical interpretations (in particular, they MUST resolve to the same biological sequence).

## Ambiguity

The spec defines "same interpretation" semantically, not lexically. Two Sequence elements with the same encoding but different literals may or may not be a conflict depending on representation (case, whitespace, alias terms).

## Decision

Default `Conservative`: warn on literal mismatch (case-sensitive byte equality). `Strict` upgrades to error. `Lenient` suppresses.

## Rationale

Most workflows treat differing literals as suspicious but not strictly a violation (the spec's "same interpretation" is a producer claim). A warning is the right default; CI gates can promote to error.

## Examples

### Conservative (default)
Matches today's validator behavior: the diagnostic shape and severity are exactly what the current test suite expects.

### Strict
(promotes the warning to an error, or activates a check that is currently suppressed.)

### Lenient
(drops the check; the rule is recorded in coverage as suppressed.)
